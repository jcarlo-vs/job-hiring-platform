# Architectural Decisions

One line per decision, with the reason. Newest at the bottom.

## Phase 0
- 2026-06-09 - **Background jobs: Inngest** (over Supabase Edge Function + DB webhook) - retries, concurrency, idempotency, and an observability dashboard out of the box; deploys alongside Next.js on Vercel.
- 2026-06-09 - **Screening model: `claude-haiku-4-5`** - cheap/fast ($1/$5 per 1M) and sufficient for structured scoring at portfolio volume; user choice.
- 2026-06-09 - **AI output via structured outputs (`output_config.format` JSON schema)** rather than prompt-only "return-only-JSON" - guarantees schema-valid output; still parse defensively and set `screening_status=ERROR` on failure as a fallback.
- 2026-06-09 - **New Supabase project** (not reusing `freelancehub`/`node-tree`) - avoids schema/data collision with two unrelated paused projects; free-tier project cost is $0.
- 2026-06-09 - **Next.js 16 + App Router + Turbopack + React 19** via create-next-app recommended defaults - aligns with Vercel's own defaults (deploy target).
- 2026-06-09 - **Tailwind v4 (CSS-first)** - `@import "tailwindcss"` + `@theme` in `globals.css`, `@tailwindcss/postcss`; no `tailwind.config.js` (v4 auto-detects content).
- 2026-06-09 - **ESLint flat config (`eslint.config.mjs`) + Prettier** with `prettier-plugin-tailwindcss` for class sorting; `next lint` is removed in Next 16, lint runs via npm script.
- 2026-06-09 - **`@supabase/ssr` (not `@supabase/auth-helpers-nextjs`)** - auth-helpers is deprecated/archived; use the getAll/setAll cookie adapter and `getUser()` (never `getSession()`) for server-side authorization.
- 2026-06-09 - **Node 22 pinned** (`.nvmrc` + engines) - `unpdf`'s bundled PDF.js v5 needs Node ≥ 22; Next 16 needs ≥ 20.9.
- 2026-06-09 - **Resume extraction: `unpdf` (PDF) + `mammoth` (DOCX)** instead of `pdf-parse` v1 - avoids the v1 `module.parent` ENOENT import bug and the native `canvas` dependency; `unpdf` ships a serverless PDF.js build. Workers run on the Node runtime, not Edge.
- 2026-06-09 - **Postgres enum types** for role/employment_type/work_mode/job_status/application_stage/screening_status/ai_recommendation - clean type-safety and good generated TS types; values can be added later via `ALTER TYPE`.
- 2026-06-09 - **Resume storage: server-generated signed upload URLs** (service-role `createSignedUploadUrl` + client `uploadToSignedUrl`) with a **private** bucket; per-bucket `allowedMimeTypes`/`fileSizeLimit` are the authoritative validation; authorization done in server routes (signed-URL generation bypasses storage RLS). Path convention `resumes/{jobId}/{applicationId}.{ext}`.
- 2026-06-09 - **RLS cross-table checks via `SECURITY DEFINER` helpers in a private schema** (`set search_path = ''`) - avoids the infinite-recursion error when a policy queries another RLS-protected table.
- 2026-06-09 - **Keep-alive: GitHub Actions cron (2×/week) hitting `/api/health`** which runs a real DB query - a static 200 does not reset Supabase's pause timer; includes a keepalive-commit so GitHub doesn't disable the schedule after 60 days.
- 2026-06-09 - **Keep-alive workflow uses no third-party Actions** - this GitHub account blocks external (non-GitHub-owned) actions ("Repository access blocked"), so the `gautamkrishnar/keepalive-workflow` action was replaced with an inline implementation using only preinstalled `curl` + `gh`; the 60-day scheduled-workflow auto-disable is mitigated by an inline gh-API empty commit guarded on >50 days of inactivity.
- 2026-06-09 - **GitHub remote uses HTTPS + the gh credential helper (repo-local)** - the machine's SSH key authenticates as a different GitHub account (`jcvs-mosaic`); origin is set to HTTPS with `credential.https://github.com.helper=!gh auth git-credential` scoped to this repo so pushes use the `jcarlo-vs` gh token without touching global git config.
- 2026-06-09 - **Kanban DnD: `@dnd-kit/core` stable line** (over `@dnd-kit/react` 0.x pre-1.0 and Atlassian pragmatic-dnd) - best accessibility + sortable preset for React 19; (applies in Phase 5).

## Phase 1
- 2026-06-09 - **Email confirmation disabled** in Supabase Auth for a frictionless demo (signup returns a session immediately). The /auth/confirm route is still built, so confirmation can be re-enabled once Resend SMTP is wired in Phase 6. (Supabase's built-in mailer is rate-limited ~2/hour, which made confirmation impractical for the demo.)
- 2026-06-09 - **Profiles created by a SECURITY DEFINER trigger** (handle_new_user) reading role + full_name from signup metadata, rather than app-side insert - atomic and robust. Clients have no profiles INSERT policy; EXECUTE on the trigger fn is revoked from API roles so it is not callable via RPC.
- 2026-06-09 - **Route protection in proxy.ts** (not per-page guards) - one place redirects unauthenticated users from protected prefixes and keeps signed-in users off the auth pages; refreshed cookies are copied onto redirect responses.

## Phase 2 (scope decisions, some land in later phases)
- 2026-06-09 - **Resume model: profile default + per-application snapshot** - applicant uploads one CV to their profile (the default); clicking Apply with no CV shows an "upload first" modal, and the upload becomes the default; on apply the file is snapshotted to resumes/{jobId}/{applicationId}.ext so past applications keep the exact file submitted (implemented Phase 3).
- 2026-06-09 - **AI screening is advisory, never auto-reject** - the worker produces score + Strong/Moderate/Weak + reasoning and moves the candidate to SCREENED; the employer makes the final accept/reject (honors the non-negotiable human-in-the-loop principle; implemented Phase 4/5).
- 2026-06-09 - **Job expiry: employer-set `expires_at`, default 30 days** - expired jobs are hidden from the public board and search, applying is blocked (enforced in the applications INSERT policy), and they appear under an "Expired" section on the employer dashboard (reopen lets the employer set a new date).

## Theme
- 2026-06-09 - **Theme: "Playful Pop"** chosen from 10 mockups - warm cream background (#fffaf3), violet primary (#9b5de5), fully rounded chunky buttons with a hard drop shadow, rounded borders, and the Nunito rounded font. Applied globally via tokens + component classes in globals.css; the /themes preview was removed.

## Phase 4
- 2026-06-10 - **Inngest v4 (4.5.0), not the v3 API** - `createFunction` is now 2-arg: `createFunction({ id, triggers: [{ event }], retries, concurrency, onFailure }, handler)`. The old 3-arg positional `(config, trigger, handler)` form is gone. Typed events use `eventType(name, { schema: staticSchema<T>() })` (usable as both a trigger and for sending); the v3 `EventSchemas` builder was removed. Verified against the installed `.d.ts`, not training data.
- 2026-06-10 - **Idempotency via an atomic DB claim, not Inngest's `idempotency` config** - the worker's first step conditionally updates `screening_status PENDING|ERROR → PROCESSING` and exits if it claims nothing. Inngest's `idempotency` key would dedupe the event for 24h and silently drop a legitimate employer re-screen; the DB claim both prevents double-processing and leaves re-screens working. Verified live (re-sent event skipped, `updated_at` unchanged).
- 2026-06-10 - **Screening output via a hand-written JSON Schema on `output_config.format`** (not the SDK's `zodOutputFormat` helper) - avoids coupling to the SDK helper's zod version (project is on zod v4); we own a small schema + defensive parse and clamp `score` to 0-100 (structured outputs ignore numeric bounds, and the DB has a 0-100 check). Contract: `{ score 0-100, recommendation STRONG|MODERATE|WEAK, summary, matched[], missing[], flags[] }`.
- 2026-06-10 - **Screening call shape: `claude-haiku-4-5`, non-streaming, max_tokens 2048, no `effort`/`thinking`** - output is small structured JSON; `effort` errors on Haiku and adaptive thinking is unnecessary for a scoring task. System prompt enforces advisory-only + bias-avoidance + calibration (most candidates MODERATE/WEAK).
- 2026-06-10 - **Worker uses the service-role admin client on the Node runtime** - all worker DB + Storage access bypasses RLS (route handlers default to Node, which unpdf/mammoth/Anthropic SDK require). Resume text extraction is `unpdf` (PDF) + `mammoth` (DOCX), capped at 40k chars; empty/scanned extraction throws `NonRetriableError`.
- 2026-06-10 - **Trigger security = Inngest request signing** - in prod the Vercel integration injects `INNGEST_SIGNING_KEY` and the SDK verifies every request, so outsiders can't invoke the worker; `/api/inngest` is deliberately not a protected proxy prefix. No custom shared-secret needed.
- 2026-06-10 - **Employer re-screen checks ownership in app code + resets via admin client** - `rescreenApplication` verifies the caller owns the application's job, resets `screening_status → PENDING`, and re-enqueues; it does not rely on an employer applications-UPDATE RLS policy (that, and the UI to call it, land in Phase 5). `stage` only ever advances `APPLIED → SCREENED`, never backward.
- 2026-06-10 - **Enqueue on apply is best-effort** - `applyToJob` wraps `inngest.send` in try/catch: the application row + resume snapshot are already committed, so a transient queue outage must not fail the apply (the row stays PENDING and an employer can re-screen).

## Phase 5
- 2026-06-10 - **No new migration - Phase 1 RLS already covers Phase 5** - `applications_select_applicant_or_employer` and `applications_update_employer` (both via `private.owns_job`) plus `resumes_select_applicant_or_employer` already let an employer read + restage their jobs' applications and open the resumes. Phase 5 is pure UI + server actions on top.
- 2026-06-10 - **Stage changes run under the employer's own session, RLS is the security boundary** - `updateApplicationStage` does a normal `update().eq(id)` as the signed-in user; `applications_update_employer` (owns_job) makes a non-owner's update match zero rows. The action also validates the stage against `PIPELINE_STAGES` (defense in depth) and pages re-check `job.employer_id === user.id`. No service role for writes.
- 2026-06-10 - **Applicant names read with the service role, after an ownership check** - `profiles` RLS only exposes the caller's own row, so the employer can't read applicants' names via their session. The applicants/detail pages verify job ownership, then use the admin client to fetch `full_name` (and to mint resume signed URLs). Reads only; authorized in app code first.
- 2026-06-10 - **Kanban: `@dnd-kit/core` only (no `@dnd-kit/sortable`)** - intra-column order isn't persisted (only `stage`), so droppable stage columns + draggable cards are enough. PointerSensor (5px activation) + KeyboardSensor for accessibility; `touch-none` on cards for mobile drag.
- 2026-06-10 - **Optimistic stage moves with shared client state (no router.refresh on move)** - `ApplicantsView` owns one `applicants` state seeded from props; both the table `<select>` and the board drag call the same `move()`, which updates state immediately and reverts on a failed action. Local state is authoritative for the page; `revalidatePath` in the action keeps other pages fresh on next load.
- 2026-06-10 - **Resume preview: service-role signed URL (300s), iframe for PDF** - PDFs render inline in an `<iframe>`; `.doc/.docx` show an "open in new tab" download link (browsers won't render them inline). The advisory-only / human-in-the-loop note sits directly under the AI panel.
- 2026-06-10 - **Applicants routes protected at the proxy** - added `/jobs/[id]/applicants(/...)` to the proxy's protected-prefix regex (next to `/edit`) so unauthenticated hits get an instant 307, instead of the page-level `redirect()` falling back to a 1s meta-refresh after the shell streams.

## Phase 6
- 2026-06-10 - **Emails via Inngest functions, not inline sends** - `send-application-received` (triggered by the existing `application/submitted`) and `send-stage-change` (triggered by a new `application/stage-changed` event emitted from `updateApplicationStage`) run as separate Inngest functions. Decoupled + retryable, and they don't block or fail the user action. Stage emails fire only on employer-initiated moves, never on the worker's automatic APPLIED->SCREENED (not a human decision, and it writes the stage directly without the event).
- 2026-06-10 - **Email degrades to a no-op without a key** - `lib/email.ts` checks `RESEND_API_KEY` (`emailEnabled()`); unset means the functions return `{skipped}` and the app runs normally. Keeps local dev and the live site working before Resend is configured.
- 2026-06-10 - **Applicant email resolved via `auth.admin.getUserById`** - the address lives in `auth.users`, not `profiles`, so the email worker (service role) reads it from the admin auth API rather than adding an email column to profiles.
- 2026-06-10 - **Branded HTML emails with inline styles** - email clients strip `<style>`/external CSS, so templates use inline styles matching the Playful Pop palette. Links use `NEXT_PUBLIC_SITE_URL` (defaults to the live URL).
- 2026-06-10 - **Landing page is a role-aware Server Component** - `app/page.tsx` reads `getProfile()` and renders CTAs per visitor (guest -> sign up / browse; employer -> post job / dashboard; applicant -> browse / my applications). No client JS needed.
- 2026-06-10 - **Accessibility: focus-visible ring in `@layer base`** - buttons, links, and dnd-kit draggable cards (`[role="button"]`) get a `2px` primary outline on keyboard focus only; inputs/selects keep their existing focus border. Targeted so it doesn't fight the `outline-none` on `.field-input` or change mouse UX.
- 2026-06-10 - **States: one root `app/error.tsx` + targeted `loading.tsx`** - a root catch-all error boundary covers every route without a closer one (`/jobs` keeps its own). Added `loading.tsx` for `/dashboard` and the `/jobs/[id]/applicants` subtree, since `loading.tsx` cascades to children and they would otherwise inherit the jobs-list skeleton.
